## Accomplishments

### Task 1 — BookingClient.tsx (commit `c35405c`)

- `handleApplyDiscount` POSTs `trip_date: selectedDate` and `trip_time: selectedTime` to `/api/discount-codes/validate` (DISC-09 client side). These are guaranteed populated by step 2 because step 1's "Continue to Details" button is `disabled={!selectedTime}` — no null guards required.
- `setDiscountAmount(data.amount)` annotated with inline `// PER-GUEST amount (DISC-07)` comment so downstream readers understand the state semantic shift.
- `calculateTotal` final return changed from `return base - discountAmount;` to `const totalDiscount = discountAmount * partySize; return Math.max(0, base - totalDiscount);` — per-guest math + floor-at-zero invariant preserved.
- `handlePaymentSuccess` `discount_amount` field changed from `discountAmount || 0` to `(discountAmount * Number(formData.party_size)) || 0` — DISC-06 fix preserved (discount_code line at 271 untouched), DISC-07 receipt fidelity established. Stored `bsp_bookings.discount_amount` now equals the dollar amount actually deducted from the Stripe charge, matching what the success page and confirmation email render.
- Both `<PriceBreakdown discountAmount=...>` invocations (step 2 around line 553, step 3 around line 612) updated to `discountAmount * (Number(formData.party_size) || 0)` — matches `calculateTotal()` and the stored receipt value.
- `PriceBreakdown.tsx` itself NOT modified — remains a pure presentation component that takes total discount.

### Task 2 — DiscountCodesClient.tsx (commit `9c915d6`)

- Form state init extended with `max_redemptions: '0'` and `excludes_early_bird: false` for both `formData` and `editData`.
- `handleAdd` payload extended with `max_redemptions: parseInt(formData.max_redemptions, 10) || 0` and `excludes_early_bird: formData.excludes_early_bird`. Form reset on success now includes the new fields.
- `handleEdit` signature widened to `(id, code_name, amount, max_redemptions, excludes_early_bird)` so the edit form pre-populates with current values.
- `handleEditSave` PATCH payload extended with the same two fields.
- Add form layout: "Amount" label changed to "Discount per Guest ($) *" with placeholder `30.00`; below it a new "Max Redemptions" input with placeholder `0 = unlimited`; below that a `md:col-span-2` row with the "Exclude Early Bird Flights" checkbox.
- Table header: "Amount" renamed to "Per Guest"; new "Redemptions" column header inserted between Per Guest and Status; loading/empty `colSpan` updated from 4 to 5.
- Row `className` now conditional: `opacity-50` applied when `code.max_redemptions > 0 && code.times_redeemed >= code.max_redemptions` (D-13 visual mute at cap).
- Redemptions cell:
  - Display mode: `{code.times_redeemed} / {code.max_redemptions > 0 ? code.max_redemptions : '∞'}` + amber "No Early Bird" pill when `excludes_early_bird` is true.
  - Edit mode: stacked max_redemptions number input + "No Early Bird" checkbox (kept compact via `space-y-2` div).
- `handleEdit` call site at the Edit button updated to pass the additional fields from the row.
- All preserved verbatim: active/inactive toggle badge with `handleToggleActive`, delete-with-confirm via `handleDelete`, inline-edit save/cancel buttons, "Add New Code" toggle button text, "$XX.XX" dollar format.

## Decisions Made

- **State semantic chosen:** Option A from RESEARCH.md Example 5 (per-guest stored, multiplied at use). Alternative (store-total) was rejected because it would require re-fetching the code or re-multiplying on every party_size change, doubling complexity without removing party_size as the source of truth.
- **Discount badge text untouched:** Plan explicitly prohibits changing the discount input UI markup (lines 497-516 of original). The applied-badge text "${discountAmount.toFixed(2)} off" is in that prohibited block. Under the new semantic this technically reads as "$30 off (per guest)" without the parenthetical — accepted per plan instruction. Operator can request a label tweak as a follow-up quick task if confusing in production.
- **Inline-edit placement:** max_redemptions input + excludes_early_bird checkbox placed in the Redemptions cell (not split into a new cell, not crowded into the Per Guest cell). Plan offered Claude's discretion here.
- **Early-bird flag rendering:** Amber pill next to the X/Y counter rather than separate column. Keeps table from sprawling to 6 columns; preserves at-a-glance scanability.
