### Locked Decisions

**Schema design**
- **D-01:** Keep existing column name `amount` on `bsp_discount_codes` but its semantics change from flat-dollar-total to dollar-per-guest. Document the semantic change with a SQL comment. No destructive rename; no production data depends on flat-total semantics.
- **D-02:** Add three new columns to `bsp_discount_codes`:
  - `max_redemptions INTEGER NOT NULL DEFAULT 0` (0 = unlimited)
  - `times_redeemed INTEGER NOT NULL DEFAULT 0`
  - `excludes_early_bird BOOLEAN NOT NULL DEFAULT false`
- **D-03:** Counter increment happens in `POST /api/bookings` after the booking row is inserted successfully, using a conditional UPDATE with `WHERE times_redeemed < max_redemptions OR max_redemptions = 0`.

**Code matching and validation**
- **D-04:** Case-insensitive matching preserved — codes normalized to uppercase at API write layer; validate endpoint uppercases input before lookup.
- **D-05:** Validate endpoint enforces in order: exists → is_active → times_redeemed < max_redemptions (when max_redemptions > 0) → if excludes_early_bird, slot is not early-bird.
- **D-06:** Slot-type detection uses `getSlotType(dateStr, timeStr)` from `src/config/solarSchedule.ts`. Validate endpoint receives `trip_date` and `trip_time` from the booking form.
- **D-07:** Error messages (exact strings, rendered directly by client):
  - Invalid/not found: `"Invalid discount code"`
  - Inactive: `"This discount code is no longer active"`
  - At cap: `"This discount code has reached its redemption limit"`
  - Early-bird conflict: `"This code cannot be used with early bird flights"` (exact wording per user)

**Pricing math**
- **D-08:** Discount = `code.amount * party_size`, floored at 0 with `Math.max(0, total - discount)`.
- **D-09:** Per-guest discount computed in TWO places that must agree: client-side `calculateTotal` in `BookingClient.tsx` for display, server-side `create-payment-intent` for Stripe authorization. Both read `party_size`.

**Redemption-limit race condition**
- **D-10:** Conditional UPDATE pattern:
  ```
  UPDATE bsp_discount_codes
  SET times_redeemed = times_redeemed + 1
  WHERE code_name = $1
    AND (max_redemptions = 0 OR times_redeemed < max_redemptions)
  RETURNING times_redeemed
  ```
  If row count is 0 (at cap), log a warning but do NOT fail the booking — Stripe has already charged.
- **D-11:** Validate endpoint does NOT decrement on cancel. Defer cancellation/refund reversal to a future phase.

**Admin UI**
- **D-12:** Add three inputs to create/edit form in `DiscountCodesClient.tsx`:
  - "Discount per guest ($)" — numeric, replaces "Amount" label
  - "Max redemptions" — integer, placeholder "0 = unlimited"
  - "Exclude early bird flights" — checkbox toggle
- **D-13:** List view shows "Redemptions" column rendered as `{times_redeemed} / {max_redemptions || '∞'}`. When at cap, row visually muted (reuse inactive style).
- **D-14:** Preserve existing inline-edit pattern, active/inactive toggle, delete-with-confirm.

**Seed data**
- **D-15:** Inline SQL seed in `bsp-schema.sql` (after new columns) using `INSERT … ON CONFLICT (code_name) DO UPDATE`:
  ```sql
  INSERT INTO bsp_discount_codes (code_name, amount, max_redemptions, times_redeemed, excludes_early_bird, is_active)
  VALUES ('30OFF', 30, 6, 0, true, true)
  ON CONFLICT (code_name) DO UPDATE SET
      amount = EXCLUDED.amount,
      max_redemptions = EXCLUDED.max_redemptions,
      excludes_early_bird = EXCLUDED.excludes_early_bird,
      is_active = EXCLUDED.is_active;
  ```
  `times_redeemed` intentionally NOT updated on conflict (preserve dev test counts).

## Summary

This phase extends the shipped Phase 2 discount-codes feature with three orthogonal capabilities: per-guest dollar amounts, a redemption-cap-with-counter, and an early-bird exclusion flag. The codebase is well-architected for this: every touch point is small, isolated, and already follows established patterns from Phase 2. There are no greenfield decisions remaining — CONTEXT.md locks every meaningful choice. The planner's job is to sequence the touch points so the schema migration lands first, server logic next, then client + admin UI in parallel, with the seed code as the closing act.

The single non-trivial technical question is the race-condition protection on `times_redeemed`. Verified: Supabase JS supports the required pattern — `.update({...}).eq(...).filter(...).select()` returns affected rows, and PostgreSQL's UPDATE…WHERE is atomic at the row level (single-statement; PostgreSQL acquires row locks during UPDATE preventing concurrent over-increment when the WHERE clause references the same counter). No RPC function needed.

**Primary recommendation:** 3 waves. Wave 1 (sequential foundation): schema migration + TypeScript interface + validate endpoint extension. Wave 2 (parallel): create-payment-intent math + bookings counter-increment + admin UI extension + client booking flow plumbing. Wave 3 (closure): seed insert SQL + smoke verification.

### Pattern 1: Schema-First Migration (already established in Phase 2)
**What:** Append idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS` blocks at end of `bsp-schema.sql`; user runs them manually in Supabase SQL Editor.
**When to use:** Always — Phase 2 and Phase 1 both followed this; verification report (`02-VERIFICATION.md`) confirms it as the standard.
**Example:**

```sql
ALTER TABLE bsp_discount_codes
    ADD COLUMN IF NOT EXISTS max_redemptions INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN IF NOT EXISTS times_redeemed INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN IF NOT EXISTS excludes_early_bird BOOLEAN NOT NULL DEFAULT false;

COMMENT ON COLUMN bsp_discount_codes.amount IS
    'Per-guest discount in dollars. Final discount = amount * party_size. (Semantics changed in Phase 02.1 — previously flat-total)';
```

### Pattern 2: Conditional Atomic UPDATE for Counter (Supabase JS)
**What:** Use `.update()` chained with `.eq()` (code_name match) and `.or()` filter (cap guard) and `.select()` to detect whether any row matched. Single statement; PostgreSQL row-locks during UPDATE.
**When to use:** Counter increment in `POST /api/bookings`, after the booking row is successfully inserted (and only if `discount_code` is non-null on the request).
**Example (verified against Supabase JS docs):**
```typescript
// Source: src/app/api/bookings/route.ts (NEW code added after line 109)
// Pattern verified at https://supabase.com/docs/reference/javascript/update
if (discount_code) {
    const normalizedCode = String(discount_code).trim().toUpperCase();
    const { data: updated, error: incErr } = await supabase
        .from('bsp_discount_codes')
        .update({ times_redeemed: supabase.rpc ? undefined : undefined })  // see note
        // NOTE: supabase-js v2 has no native increment helper; use raw SQL via RPC OR a read-then-write
        // RECOMMENDED PATH (no RPC needed): use a one-liner update + select via a manual SQL via .rpc('sql', ...) — NOT available.
        // CORRECT PATTERN: use supabase.from(...).update({ times_redeemed: <new_value> }).eq(...).gte(...) — but this requires reading the row first OR using a stored-procedure.
        // SEE BELOW: planner must use the .rpc() route OR a 2-step "select-for-update style" approximation.
```

**REVISED — CRITICAL CLARIFICATION FOR PLANNER:** `supabase-js` v2 (`@supabase/supabase-js@2.106.2`) does **not** ship a column-arithmetic syntax (e.g. `{ times_redeemed: 'times_redeemed + 1' }` is NOT supported and will be coerced to a string literal, breaking the schema). There are TWO viable patterns:

**Pattern 2A — RPC stored function (recommended, gives true atomicity):**
```sql
-- Add to supabase/bsp-schema.sql §10 after the new columns
CREATE OR REPLACE FUNCTION bsp_increment_discount_redemption(p_code_name TEXT)
RETURNS TABLE(times_redeemed INTEGER, at_cap BOOLEAN) AS $$
DECLARE
    v_updated_count INTEGER;
    v_new_count INTEGER;
BEGIN
    UPDATE bsp_discount_codes
    SET times_redeemed = times_redeemed + 1,
        updated_at = now()
    WHERE code_name = p_code_name
      AND (max_redemptions = 0 OR times_redeemed < max_redemptions)
    RETURNING bsp_discount_codes.times_redeemed INTO v_new_count;

    GET DIAGNOSTICS v_updated_count = ROW_COUNT;

    IF v_updated_count = 0 THEN
        -- At cap; do not throw — return at_cap=true so caller can log
        RETURN QUERY SELECT NULL::INTEGER, true;
    ELSE
        RETURN QUERY SELECT v_new_count, false;
    END IF;
END;
$$ LANGUAGE plpgsql;
```
Call site:
```typescript
const { data: incData, error: incErr } = await supabase.rpc('bsp_increment_discount_redemption', {
    p_code_name: String(discount_code).trim().toUpperCase(),
});
if (incErr) {
    console.error('[DISCOUNT] Counter increment failed (booking already saved):', incErr);
} else if (incData?.[0]?.at_cap) {
    console.warn('[DISCOUNT] Code at cap on increment — Stripe already charged:', discount_code);
}
```

**Pattern 2B — read-then-conditional-update (no SQL function; race-window exists but is < 100ms):**
```typescript
// 1. Read current counter and cap
const { data: codeRow } = await supabase
    .from('bsp_discount_codes')
    .select('id, times_redeemed, max_redemptions')
    .eq('code_name', normalizedCode)
    .single();

// 2. Conditional UPDATE guarded by .lt() so race-condition increment fails atomically at the DB level
const newCount = (codeRow?.times_redeemed ?? 0) + 1;
const { data: updated, error: incErr } = await supabase
    .from('bsp_discount_codes')
    .update({ times_redeemed: newCount, updated_at: new Date().toISOString() })
    .eq('id', codeRow.id)
    .eq('times_redeemed', codeRow.times_redeemed)  // optimistic-concurrency guard
    .select();

if (!updated || updated.length === 0) {
    console.warn('[DISCOUNT] Race lost on counter increment (or at cap) — Stripe already charged:', normalizedCode);
}
```

**Planner choice:** Pattern 2A (RPC function) is the safer and idiomatic approach for D-10. It executes the conditional UPDATE in a single SQL statement using PostgreSQL's row-level locking — true atomicity. Pattern 2B is the fallback if the user prefers not to add a SQL function. **Recommended: 2A.** Both must be paired with the same `WHERE max_redemptions = 0 OR times_redeemed < max_redemptions` guard at the validate and payment-intent layers as a fast-fail.

### Pattern 3: Validate Endpoint Backward-Compatible Param Extension
**What:** Add `trip_date` and `trip_time` to the request body of `POST /api/discount-codes/validate`, but treat them as optional with a defensive default: if missing AND code has `excludes_early_bird=true`, allow the validation through (don't 400). This keeps callers that haven't been updated from breaking.
**When to use:** Any param addition where existing callers might not be updated.
**Example:**

### Anti-Patterns to Avoid
- **String-arithmetic in supabase-js update:** `{ times_redeemed: 'times_redeemed + 1' }` — does NOT work; supabase-js v2 stringifies the value. Use the RPC pattern or read-then-write.
- **Failing the booking when counter increment fails:** The Stripe charge has already completed before the counter increments. Failing the booking after a successful charge would leave the customer charged with no booking row. Log the warning and proceed.
- **Skipping the cap check at create-payment-intent:** Even though validate also checks the cap, a customer can hold a stale "applied" state on the client and bypass validate-time checks. The payment-intent layer is the last gate before charging.
- **Hardcoding "30OFF" anywhere outside the seed SQL:** No code reference to the literal string — it's a runtime DB row only.
- **Re-uppercasing on every read:** The DB stores codes uppercase already (Phase 2 invariant in `route.ts:36`); only normalize at the write boundary and in validate input.

## Common Pitfalls

### Pitfall 1: supabase-js cannot increment in-place
**What goes wrong:** Developer writes `.update({ times_redeemed: 'times_redeemed + 1' })`. The string `'times_redeemed + 1'` is sent to PostgREST and parsed as a literal; PostgreSQL rejects it as a type mismatch (or worse, stores it if the column were TEXT).
**Why it happens:** Familiar with SQL but unfamiliar with PostgREST's JSON-only update body. PostgREST does not parse expressions.
**How to avoid:** Use the RPC function (Pattern 2A) OR the read-then-write pattern (Pattern 2B). Both are documented and verified.
**Warning signs:** TypeScript error on the value type (number expected, string given); 400 from PostgREST; counter doesn't change.

### Pitfall 2: Stale client state allows cap-exceeded charges
**What goes wrong:** Customer applies a valid code at validate-time (counter was 5/6). Two other customers complete bookings before this one. Customer A's stale client still believes the code is valid; they reach `create-payment-intent`, which doesn't re-check the cap.
**Why it happens:** Validate is a snapshot; the only true authority is the moment Stripe charges.
**How to avoid:** Re-query the discount code in `create-payment-intent` (it already does, line 62-66) AND add the cap check there. If at cap at payment-intent time, silently drop the discount (per existing leniency pattern in STATE.md: "Payment intent creation is lenient on invalid discount codes — no rejection, just no discount applied"). Customer pays full price.
**Warning signs:** `times_redeemed > max_redemptions` in DB after a busy period.

### Pitfall 3: Race condition between insert and counter increment
**What goes wrong:** Booking inserted → counter increment attempted → cap was just reached by another booking → counter not incremented → DB shows N bookings but `times_redeemed = N - 1`.
**Why it happens:** Counter increment is a separate statement from booking insert. PostgreSQL doesn't (and shouldn't) treat these as one transaction.
**How to avoid:** Accept this as known-and-tolerated per D-10 and D-11. Log warning, do not retry, do not fail. Operator can reconcile via admin UI by manually adjusting `times_redeemed`.
**Warning signs:** Operator notices `times_redeemed` lower than actual usage count.

### Pitfall 4: BookingClient handleApplyDiscount missing new params
**What goes wrong:** User selects time slot in step 1, advances to step 2, enters code, clicks Apply. Without sending `trip_date` and `trip_time`, the validate endpoint cannot perform the early-bird check, so DISC-09 silently fails (early-bird codes incorrectly accepted).
**Why it happens:** The handler at `BookingClient.tsx:199-203` currently sends only `{ code: discountInput.trim() }`. Easy to miss the param addition.
**How to avoid:** Plan the BookingClient edit as the same task as the validate endpoint extension — they must ship together. State availability is fine: `selectedDate` and `selectedTime` are populated by step 2 (gated by `disabled={!selectedTime}` button on step 1).
**Warning signs:** Manual test: create code with `excludes_early_bird=true`, try to apply at 10 AM slot, code accepts silently.

### Pitfall 5: Admin form serialization mismatch
**What goes wrong:** UI sends `max_redemptions: "6"` (string from input) but POST handler at `/api/discount-codes/route.ts:32-34` only validates `amount`. The string lands in the DB column; INTEGER coerces or rejects.
**Why it happens:** Existing pattern uses `parseFloat(formData.amount)` (line 30); the new fields need analogous `parseInt(formData.max_redemptions, 10)`.
**How to avoid:** Mirror the existing parsing pattern. Add server-side validation in `route.ts:POST` for the new fields (type: number, ≥ 0 for max_redemptions; boolean for excludes_early_bird).
**Warning signs:** TypeScript error at runtime; DB constraint rejection; or silently-coerced 0 in the DB.

### Pitfall 6: Seeding before columns exist
**What goes wrong:** Operator runs only the bottom of `bsp-schema.sql` (the seed) without running the column additions first; INSERT fails.
**Why it happens:** Schema file is run-once-at-a-time; ordering matters.
**How to avoid:** Place the seed INSERT physically AFTER the `ALTER TABLE … ADD COLUMN IF NOT EXISTS` lines in the same SQL block. Document at top of §10 that the whole block must be run as one statement-batch.
**Warning signs:** SQL error `column "max_redemptions" does not exist`.

### Pitfall 7: Forgetting to extend the PATCH handler
**What goes wrong:** Admin edit form sends `max_redemptions` and `excludes_early_bird` in the PATCH body, but `/api/discount-codes/[id]/route.ts:11` only destructures `{ code_name, amount, is_active }`. New fields silently dropped.
**Why it happens:** Easy to extend only POST (create) and forget PATCH (edit).
**How to avoid:** Plan both `route.ts` and `[id]/route.ts` updates as a single task or sibling tasks in the same wave.
**Warning signs:** Editing a code's max_redemptions in admin shows the change in the UI (RTK Query optimistic cache), then a refresh reverts it.

```typescript
            const slot = getSlotType(trip_date, trip_time);
            if (slot === 'earlybird') {
                console.log('[DISCOUNT] Code excluded for early bird slot:', normalized, trip_date, trip_time);
                return NextResponse.json(
                    { valid: false, error: 'This code cannot be used with early bird flights' },
                    { status: 200 }
                );
            }
        }

        console.log('[DISCOUNT] Valid code applied:', normalized, 'amount per guest:', data.amount);

            { valid: true, code_name: data.code_name, amount: data.amount },
```

### Example 10: Schema migration block (DISC-08, DISC-11)
```sql
-- Source: append to supabase/bsp-schema.sql AFTER line 222, BEFORE line 224 (§11 heading)

-- ============================================================
-- 10b. DISCOUNT CODE PER-GUEST + CAP + EARLY-BIRD EXCLUSION (Phase 02.1)
-- Semantics of bsp_discount_codes.amount changes from flat-total to per-guest dollars.
-- Safe to re-run: ADD COLUMN IF NOT EXISTS is idempotent.
-- ============================================================
ALTER TABLE bsp_discount_codes
    ADD COLUMN IF NOT EXISTS max_redemptions INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN IF NOT EXISTS times_redeemed INTEGER NOT NULL DEFAULT 0,
    ADD COLUMN IF NOT EXISTS excludes_early_bird BOOLEAN NOT NULL DEFAULT false;

COMMENT ON COLUMN bsp_discount_codes.amount IS
    'Per-guest discount in dollars. Final discount = amount * party_size. (Semantics changed Phase 02.1 — was flat-total in Phase 2.)';
COMMENT ON COLUMN bsp_discount_codes.max_redemptions IS
    'Maximum redemptions allowed. 0 = unlimited.';
COMMENT ON COLUMN bsp_discount_codes.times_redeemed IS
    'Counter incremented atomically by bsp_increment_discount_redemption() after successful booking insert.';

-- Atomic counter increment with cap guard (Pattern 2A).
-- Returns at_cap=true if the row was at-or-above max_redemptions and no increment occurred.
CREATE OR REPLACE FUNCTION bsp_increment_discount_redemption(p_code_name TEXT)
RETURNS TABLE(times_redeemed INTEGER, at_cap BOOLEAN) AS $$
DECLARE
    v_updated_count INTEGER;
    v_new_count INTEGER;
BEGIN
    UPDATE bsp_discount_codes
    SET times_redeemed = bsp_discount_codes.times_redeemed + 1,
        updated_at = now()
    WHERE bsp_discount_codes.code_name = p_code_name
      AND (bsp_discount_codes.max_redemptions = 0
           OR bsp_discount_codes.times_redeemed < bsp_discount_codes.max_redemptions)
    RETURNING bsp_discount_codes.times_redeemed INTO v_new_count;

    GET DIAGNOSTICS v_updated_count = ROW_COUNT;

    IF v_updated_count = 0 THEN
        RETURN QUERY SELECT NULL::INTEGER, true;
    ELSE
        RETURN QUERY SELECT v_new_count, false;
    END IF;
END;
$$ LANGUAGE plpgsql;

-- Seed: "30OFF" inaugural code (DISC-11). ON CONFLICT preserves dev test counters.
INSERT INTO bsp_discount_codes (code_name, amount, max_redemptions, times_redeemed, excludes_early_bird, is_active)
```

## Open Questions

1. **Should per-guest discount apply to observers too, or only flyers (party_size)?**
   - What we know: CONTEXT D-08 says `code.amount * party_size`. `party_size` in BookingClient is the count of flyers, NOT observers. Observers come from `formData.add_ons.observer_package` which is separate.
   - What's unclear: User's "per flyer" terminology in phase name suggests flyers only. But a 4-person observer-only boat (party_size=0) would get zero discount — is that the intent?
   - Recommendation: Apply to `party_size` (flyers) only. This matches the locked decision verbatim. Document in code comment: `// DISC-07: per-flyer (party_size), not observers — per CONTEXT D-08`. If user wants observers too in future, that's a one-line change.

2. **Decimal handling for max_redemptions integer parse**
   - What we know: HTML `<input type="number">` returns string; `parseInt(str, 10) || 0` handles empty/invalid.
   - What's unclear: User entering "1.5" — parseInt yields 1 (truncates). Is silent truncation acceptable, or should admin form validate integer-only?
   - Recommendation: Add `step="1"` and `min="0"` on the input element to constrain UI. parseInt fallback is fine.

3. **Should validate endpoint reject codes when `trip_date`/`trip_time` missing AND `excludes_early_bird=true`?**
   - What we know: Pattern 3 in this doc suggests defensive default (allow through when params missing). CONTEXT D-05 doesn't specify.
   - Recommendation: Allow through (current Pattern 3) — the create-payment-intent layer is the authoritative gate and re-checks. This avoids breaking any caller that hasn't been updated. **But:** since this phase updates the only caller (BookingClient), the planner could also choose to make params required and 400 on missing. Either is defensible; pick the lenient option for forward-compatibility.

4. **Should counter increment happen on `status='confirmed'` only, or any successful insert?**
   - What we know: D-03 says "after the booking row is inserted successfully." Current code at `bookings/route.ts:98` sets `status: paymentStatus` which is `'confirmed'` for real Stripe charges and for mock-mode test bookings (line 58).
   - Recommendation: Increment on any successful insert (mock-mode included). The user's mental model in CONTEXT specifics ("operationally redemptions are tracked even in test bookings; do NOT add a 'test mode' that exempts counting. The user will manually reset times_redeemed via admin UI or SQL if needed after testing.") explicitly supports this. No conditional on `status`.

5. **Admin UI: should `times_redeemed` be editable inline?**
   - What we know: CONTEXT specifies a "Redemptions" column display (D-13) and existing inline-edit pattern (D-14). User mentioned manually resetting after testing.
   - What's unclear: Whether reset is via inline edit or a separate "Reset counter" button.
   - Recommendation: Allow `times_redeemed` to be PATCH-able through the existing `/api/discount-codes/[id]` route (accept the field). UI exposure can be Claude's discretion — a small "Reset" button next to the counter is the lowest-friction option. Falls under D-14 "preserve existing inline-edit pattern" — extend the edit form with this field hidden behind a button.
