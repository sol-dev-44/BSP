# Phase 02.1: Extend Discount Codes - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning
**Source:** Conversation with user (extends completed Phase 2)

<domain>
## Phase Boundary

Extend the existing discount codes feature (shipped in Phase 2, 2026-04-01) to support per-guest discount amounts, redemption limits, and early-bird slot exclusions. Admin UI is updated to surface these new controls. The first real-world code "30off" is seeded as the inaugural usage.

In scope:
- Schema extensions on `bsp_discount_codes` (per-guest semantics, redemption cap and counter, early-bird exclusion flag)
- Per-guest discount math in payment-intent creation and PriceBreakdown UI
- Redemption-limit enforcement at validate and payment-intent layers, with atomic counter increment after successful booking
- Early-bird rejection with exact error string at the validate endpoint
- Admin UI updates: input fields for per-guest amount, max redemptions, early-bird toggle; list display of times_redeemed / max_redemptions
- Seed migration for code "30off"

Out of scope (deferred):
- Percentage-based discounts
- Per-customer redemption limits (e.g., "one per customer")
- Date-range validity windows
- Stacking multiple codes per booking

</domain>

<decisions>
## Implementation Decisions

### Schema design
- **D-01:** Keep the existing column name `amount` on `bsp_discount_codes` but its semantics change from flat-dollar-total to dollar-per-guest. Document the semantic change with a SQL comment. (Avoids a destructive rename; no production data depends on flat-total semantics since the feature is freshly shipped and unused beyond dev tests.)
- **D-02:** Add three new columns to `bsp_discount_codes`:
  - `max_redemptions INTEGER NOT NULL DEFAULT 0` — 0 means unlimited (preserves backward compatibility for any test codes)
  - `times_redeemed INTEGER NOT NULL DEFAULT 0` — incremented after successful booking insert
  - `excludes_early_bird BOOLEAN NOT NULL DEFAULT false`
- **D-03:** Counter increment happens in the bookings API route (`POST /api/bookings`) after the booking row is inserted successfully, using a Supabase RPC or direct UPDATE with `times_redeemed = times_redeemed + 1` and a `WHERE times_redeemed < max_redemptions OR max_redemptions = 0` guard to avoid races at the cap.

### Code matching and validation
- **D-04:** Case-insensitive matching preserved as-is — codes are normalized to uppercase at the API write layer (existing behavior from Phase 2 PROJECT.md decision); validate endpoint uppercases input before lookup.
- **D-05:** Validate endpoint enforces (in order): exists → is_active → times_redeemed < max_redemptions (when max_redemptions > 0) → if excludes_early_bird, slot is not early-bird.
- **D-06:** Slot-type detection uses `getSlotType(dateStr, timeStr)` from `src/config/solarSchedule.ts` — the validate endpoint receives `trip_date` and `trip_time` from the booking form to perform this check.
- **D-07:** Error messages are precise strings the client renders directly:
  - Invalid/not found: `"Invalid discount code"`
  - Inactive: `"This discount code is no longer active"`
  - At cap: `"This discount code has reached its redemption limit"`
  - Early-bird conflict: `"This code cannot be used with early bird flights"` (exact wording per user)

### Pricing math
- **D-08:** Discount amount = `code.amount * party_size`, floored at 0 with `Math.max(0, total - discount)` to prevent negative Stripe charges (preserves existing safety pattern from Phase 2).
- **D-09:** Per-guest discount is computed in two places that must agree:
  1. Client-side `calculateTotal` in `BookingClient.tsx` for display
  2. Server-side `create-payment-intent` route for Stripe authorization amount
  Both read `party_size` from form state / request body.

### Redemption-limit race condition
- **D-10:** Use a conditional UPDATE for the counter increment so two concurrent bookings can't both push `times_redeemed` past `max_redemptions`. Pattern: `UPDATE bsp_discount_codes SET times_redeemed = times_redeemed + 1 WHERE code_name = $1 AND (max_redemptions = 0 OR times_redeemed < max_redemptions) RETURNING times_redeemed`. If the row count is 0, the code was at cap — log a warning but do not fail the booking (Stripe has already charged at this point per existing flow).
- **D-11:** Validate endpoint does NOT decrement on cancel — cancellations are a separate concern, and current Phase 2 verification report does not track refund/cancel discount reversal. Defer to a future phase.

### Admin UI
- **D-12:** Add three inputs to the create/edit form in `DiscountCodesClient.tsx`:
  - "Discount per guest ($)" — numeric, replaces the current "Amount" label
  - "Max redemptions" — integer with placeholder "0 = unlimited"
  - "Exclude early bird flights" — checkbox toggle
- **D-13:** List view shows a "Redemptions" column rendered as `{times_redeemed} / {max_redemptions || '∞'}`. When `times_redeemed >= max_redemptions > 0`, the row is visually muted (existing inactive style reused).
- **D-14:** Existing inline-edit pattern preserved; existing active/inactive toggle preserved; existing delete-with-confirm preserved.

### Seed data
- **D-15:** Inline SQL seed in `bsp-schema.sql` (after the new columns) using `INSERT ... ON CONFLICT (code_name) DO UPDATE` so re-running the schema doesn't fail and so dev/prod can both initialize the code:
  ```sql
  INSERT INTO bsp_discount_codes (code_name, amount, max_redemptions, times_redeemed, excludes_early_bird, is_active)
  VALUES ('30OFF', 30, 6, 0, true, true)
  ON CONFLICT (code_name) DO UPDATE SET
      amount = EXCLUDED.amount,
      max_redemptions = EXCLUDED.max_redemptions,
      excludes_early_bird = EXCLUDED.excludes_early_bird,
      is_active = EXCLUDED.is_active;
  ```
  Note: `times_redeemed` is intentionally NOT updated on conflict — preserves dev test counts when re-running.

### Claude's Discretion
- Exact admin UI layout/spacing for the three new fields (follow existing form patterns)
- Whether to surface a "remaining" badge vs raw counter on the codes list
- Loading/error UI for the validate endpoint when network is slow
- TypeScript interface updates (DiscountCode in discountCodesApi.ts) — derive from schema

</decisions>

<specifics>
## Specific Ideas

- User explicitly stated the error string: **"This code cannot be used with early bird flights"** — use verbatim
- User's mental model: "6 in stock, I'll test with 2 and 4 will be left over for production" — operationally this means redemptions are tracked even in test bookings; do NOT add a "test mode" that exempts counting. The user will manually reset times_redeemed via admin UI or SQL if needed after testing.
- Treat per-guest as the default and only mode — no mixed-mode (flat vs per-guest) codes. All codes are per-guest going forward.

</specifics>
