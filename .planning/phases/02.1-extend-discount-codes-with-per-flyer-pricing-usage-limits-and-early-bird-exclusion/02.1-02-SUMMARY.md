## Leniency Preservation (payment-intent)

The discount block in `create-payment-intent/route.ts` contains zero `status: 4` responses. All four failure branches (not found, inactive, at-cap, early-bird-excluded) log a `console.warn` and fall through to the default `discountAmount = 0` — Stripe authorization proceeds at full price. Matches the Phase 2 STATE.md decision: "Payment intent creation is lenient on invalid discount codes — no rejection, just no discount applied."

## Decisions Made

- Leniency extended to the two new gates (at-cap, early-bird-excluded) in payment-intent — matches Phase 2 pattern for invalid/inactive codes; validate endpoint is the user-facing gate.
- POST insertPayload uses conditional inclusion (only set when `!== undefined`) so omitted fields receive DB defaults (`max_redemptions=0`, `excludes_early_bird=false`) rather than `null`.
- PATCH accepts `times_redeemed` as a flat field rather than gating it behind a separate `/reset-counter` endpoint — keeps admin reset trivial and matches the Open Question 5 resolution from RESEARCH.md.
